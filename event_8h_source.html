83/*
84 Current system:
85 - macros inside yoyo_c_api.h are used to define custom behavior which must be implemented as a standalone function with specific sig for entry.c to register and call
86 Limitations of current system:
87 - entry.c managing some engine logic
88 - confusing for user (have to touch header to enable)
89 - functionally limited (no way to subscribe multiple functions to an event without explicitly trickling down)
90
91 Proposed system:
92 - event system that allows for multiple functions to be subscribed to an event
93 Challenges:
94 - potentially allowing persistant events
95 - events like pre init are meant to be called before the engine is initialized, so how do we handle that? - potentially having this be a standalone system that does not need the engine initialized to fire
96 - entry.c can call the engine fire event function itself
97
98 should the event system let the user touch it? like can they add their own events with void * data?
99
100 Problems:
101 - some callbacks have specific args... dont want to force user to use varargs
102*/
